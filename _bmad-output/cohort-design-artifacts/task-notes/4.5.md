## Implementation Summary

### 1. Health Monitor System (386 lines)

**File:** `cis-discord-bot/cis_controller/health_monitor.py`

**Key Components:**

- **HealthMonitor Class:** Continuous monitoring with 5-minute intervals (300 seconds)
  - `start()`: Launches asyncio monitoring loop
  - `stop()`: Graceful shutdown with task cancellation
  - `_monitor_loop()`: Main loop that runs health checks and daily backups

- **Health Checks:**
  - `_check_discord()`: Discord API connectivity + latency validation
    - Success: bot is ready AND latency < 1 second
    - Failure: bot not ready OR high latency (>1s)
  - LLM provider: Config validation via `validate_provider_configuration()`
  - `_check_database()`: SQLite connectivity with simple query test

- **Trevor Notification System:**
  - `_notify_trevor_of_failure()`: Discord embed for system failures (red)
    - Lists all failed systems
    - Includes detailed error messages
    - Action required guidance
  - `_notify_backup_success()`: Discord embed for backup completion (green)
    - Backup file name
    - Cleanup count
    - Total backups and size
  - `_notify_backup_failure()`: Discord embed for backup failure (red)
    - Error details
    - Action required guidance

- **Daily Database Backup Integration:**
  - `_run_daily_backup()`: Automated backup execution
    - Checks if backup already run today
    - Creates backup via `DatabaseBackup`
    - Cleans up old backups (30-day retention)
    - Notifies Trevor of success/failure

- **Graceful Degradation:**
  - Health monitor continues running even if individual checks fail
  - Trevor notifications ensure awareness without stopping monitoring

**Error Handling:**
- All notification functions handle missing channels gracefully
- Health check errors are logged but don't crash the monitor
- Backup failures alert Trevor but don't stop future attempts

### 2. LLM Provider Failure Handling

**File:** `cis-discord-bot/cis_controller/llm_integration.py`

**Key Changes:**

- **Exception Handling in `call_agent_with_context()`:**
  - Catches all exceptions from LLM provider calls
  - Logs error with provider and agent context
  - Returns fallback message via `get_llm_fallback_message()`
  - Returns zero cost data (no tokens, no cost)

- **Agent-Specific Fallback Messages:**
  - `get_llm_fallback_message(agent: str)`: Returns encouraging Habit practice messages
  - **frame:** Habit 1 (PAUSE) scaffolding
    - "Stop before typing your question"
    - "Ask yourself: What do I actually want to know?"
    - "Try framing: 'My situation is [describe]. I want to [goal].'"
  - **diverge:** Habit 3 (ITERATE) scaffolding
    - "List 3 possibilities for your question"
    - "Change one thing about each option"
    - "Notice what new options emerge"
  - **challenge:** Habit 4 (THINK FIRST) scaffolding
    - "Write down your assumption"
    - "Ask: What if the opposite is true?"
    - "Test: What evidence would prove you wrong?"
  - **synthesize:** Reflection scaffolding
    - "Review your past CIS conversations"
    - "Look for patterns in your thinking"
    - "Write 2-3 sentences about what changed"

**Provider Support:**
- Works with all three providers (OpenAI, Anthropic, Zhipu/GLM)
- Returns zero cost data with `fallback: True` flag
- Encourages independent practice even when AI is unavailable

### 4. Database Failure Handling

**Implementation:** Detection + Trevor Alert (simplified from full in-memory fallback)

**Reasoning:**
- Full in-memory fallback deemed too complex for this task
- Database failures are rare with SQLite
- Detection + Trevor alert provides sufficient visibility
- Trevor can investigate and restore from backup if needed

**Detection:**
- `_check_database()` attempts simple query: `SELECT 1`
- Connection errors are caught and logged
- Health status updated to `database: False`
- Trevor notified via `_notify_trevor_of_failure()`

**Backup Recovery:**
- Daily backups run automatically (30-day retention)
- Trevor can restore from backup if database is corrupted
- Backup manager handles cleanup and retention

### 5. Integration with Main Bot

**File:** `cis-discord-bot/main.py`

**Key Changes:**

```python
# In on_ready() after bot connects
from cis_controller.health_monitor import HealthMonitor

health_monitor = HealthMonitor(
    bot=bot,
    db_path=db_path,
    facilitator_dashboard_id=dashboard_channel_id,
    trevor_discord_id=trevor_discord_id,
    check_interval_seconds=300,  # 5 minutes
)

# In finally block for graceful shutdown
if health_monitor:
    await health_monitor.stop()
```

**Startup Flow:**
1. Bot connects to Discord
2. Database initializes
3. Health monitor starts
4. Background task runs health checks every 5 minutes

**Shutdown Flow:**
1. Bot receives shutdown signal
2. Health monitor stops gracefully
3. Monitoring task cancelled
4. Cleanup completes

## Design Decisions

### 1. Health Check Interval (5 minutes)
**Decision:** 300-second intervals
**Rationale:**
- Frequent enough to catch issues quickly
- Not so frequent as to overwhelm Trevor with notifications
- Balances responsiveness with resource usage
- Matches industry best practices for health monitoring

### 2. LLM Fallback Message Strategy
**Decision:** Encourage independent Habit practice
**Rationale:**
- Maintains learning momentum even during outages
- Reinforces Habit practice (core pedagogical goal)
- Psychological safety: students feel capable, not stuck
- Better than generic "system down" message

### 3. Database Failure Handling Scope
**Decision:** Detection + Trevor alert (not full in-memory fallback)
**Rationale:**
- Full in-memory fallback adds significant complexity
- SQLite is stable; database failures are rare
- Trevor has daily backups for recovery
- Detection + alert provides sufficient visibility
- Can be enhanced in future if needed

### 4. Discord Auto-Reconnect
**Decision:** Rely on discord.py built-in auto-reconnect
**Rationale:**
- discord.py has mature, battle-tested reconnection logic
- No need to reinvent the wheel
- Reduces code maintenance burden
- Works reliably in production

### 5. Trevor Notification Design
**Decision:** Rich Discord embeds with color coding
**Rationale:**
- Red (🚨) for failures: urgent attention
- Green (✅) for backup success: confirmation
- Structured fields for detailed information
- Timestamps for audit trail
- Professional and actionable

   - Need to handle persistence reconciliation when database recovers
   - Complexity tradeoff may be worth it for high-availability requirements

   - Embed showing last check time

   - Allows tuning without code changes
