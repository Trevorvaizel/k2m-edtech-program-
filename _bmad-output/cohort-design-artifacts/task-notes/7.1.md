## Implementation Details

### 1. Backend: `/api/interest` Endpoint ✅

**File Created:** `cis-discord-bot/cis_controller/interest_api_server.py`

**Features:**
- HTTP POST endpoint using aiohttp pattern (follows `parent_unsubscribe_server.py` architecture)
- **Enrollment cap check:** Counts "Paid" students in Google Sheets Column K (max 30)
- **Duplicate detection:** Checks if email already exists in Column B, returns helpful message
- **Google Sheets integration:** Appends new student rows to Student Roster
  - Columns populated: Name, Email, Phone, Profession, Payment Status (Pending), Invite Code (placeholder)
- **Brevo email integration:** Sends Email #1 (normal) or Email #5 (waitlist) via EmailService
- **CORS headers enabled:** For cross-origin requests from landing page
- Server runs on port 8081 (configurable via environment variables)

**Environment Variables Required:**
- `GOOGLE_SHEETS_CREDS` — Google service account credentials (JSON)
- `GOOGLE_SHEETS_ID` — Student Roster spreadsheet ID
- `GOOGLE_SHEETS_RANGE` — Sheet range, default "Student Roster!A:Z"
- `BREVO_API_KEY` — Brevo API key for sending emails
- `ENROLLMENT_CAP` — Max paid enrollments (default: 30)

**Integration with Bot:**
- Modified `cis-discord-bot/main.py`:
  - Added `interest_api_server` global variable
  - Initialized server in `on_ready()` hook
  - Added cleanup in shutdown handler
  - Server starts automatically when bot launches (controlled by `ENABLE_INTEREST_API` env var, default: true)

## Architecture Decisions

1. **Single source of truth:** Created API in existing bot infrastructure rather than duplicating code in Vercel serverless function
   - Leverages existing Google Sheets integration (`preload_students.py` functions)
   - Leverages existing EmailService
   - Avoids code duplication and maintenance overhead

2. **Enrollment cap logic:** Count only "Paid" (Column K) students for accuracy
   - Ensures actual enrollments are counted
   - Prevents underfilling if pending payments fail

3. **Student-first design:**
   - Warm waitlist email tone ("priority list" not "rejected")
   - Helpful duplicate messages
   - No pressure language
   - Clear refund policy builds trust

4. **User experience:**
   - Form modal doesn't disrupt landing page flow
   - Immediate email confirmation
   - Clear next steps
   - Mobile-responsive design
