✅ Task 1.2 COMPLETE - StudentContext + SQLite schema (2026-02-17)

**Tech Stack Confirmed (Party Mode Decision):**
- SQLite for 2-3 cohorts (400-600 concurrent students)
- One database file per cohort (cohort-1.db, cohort-2.db)
- PostgreSQL migration at 4+ cohorts
- MongoDB rejected: adds complexity, no benefit for relational data

**Files Created:**
1. database/backup.py (370 lines):
   - DatabaseBackup class with automated backups
   - 30-day retention policy (configurable)
   - Multi-cohort support (backup_all_cohorts)
   - SQLite online backup API (reliable, handles concurrent access)
   - Backup verification and cleanup

2. database/migration.py (390 lines):
   - DatabaseMigration class (SQLite → PostgreSQL)
   - Transaction safety (rollback on failure)
   - Schema + data migration in batches (1000 rows/batch)
   - Type conversions (SQLite → PostgreSQL compatible)
   - Index recreation after migration
   - Verification (row count comparison)
   - Multi-cohort migration (migrate_all_cohorts)

3. database/db_cli.py (390 lines):
   - Unified CLI for all database operations:
   * backup: Single or all cohorts
   * restore: From backup files
   * status: Database + backup info
   * migrate: SQLite → PostgreSQL
   * migrate-all: All cohorts to PostgreSQL
   - Argument parsing with --help, examples
   - Error handling and exit codes

4. tests/test_database.py (420 lines):
   - 8 test classes, 25+ test cases:
   * TestStudentCRUD: create/get/update students, Discord ID normalization
   * TestConversationCRUD: save/retrieve conversation history
   * TestHabitPracticeCRUD: habit tracking initialization and updates
   * TestObservabilityEvents: event logging with privacy hashing
   * TestStudentContextORM: build_student_context factory, altitude calculation
   * TestDatabaseIndexes: verify all 8 indexes exist, performance tests
   * TestMultiCohortSupport: separate DB files per cohort, concurrent access
   * TestDataIntegrity: foreign keys, unique constraints, ON DELETE CASCADE

**Indexes Verified (schema.sql:26-28, 47, 66-68, 123-125):**
- students: cohort_id, current_week, zone (3 indexes)
- habit_practice: student_id (1 index)
- conversations: student_id, (student_id, agent), created_at (3 indexes)
- observability_events: student_id_hash, event_type, timestamp (3 indexes)
- Total: 10 indexes optimized for 400+ concurrent students

**Multi-Cohort Architecture:**
- Separate database files: cohort-1.db, cohort-2.db, etc.
- Zero contention between cohorts
- Isolated backups (cohort-1.db.backups/, cohort-2.db.backups/)
- Parallel migrations to PostgreSQL schemas (cohort_1, cohort_2)
