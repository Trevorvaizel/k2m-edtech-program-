## What Was Built

### 1. DB Column: `token_warning_sent` (schema_pg.sql + pg_store.py)

- Added `token_warning_sent BOOLEAN DEFAULT FALSE` to the `students` table in `database/schema_pg.sql`
- Added idempotent migration: `ALTER TABLE students ADD COLUMN IF NOT EXISTS token_warning_sent BOOLEAN DEFAULT FALSE`
- Added two methods to `database/pg_store.py`:
  - `get_token_warning_sent(enrollment_email)` → bool
  - `set_token_warning_sent(enrollment_email, value: bool)` → None
- Lookup is by `enrollment_email` (not `discord_id`) so students who haven't joined Discord yet are also trackable

### 2. Nightly Scheduler Job (scheduler/scheduler.py)

- Added `_token_expiry_check_today` flag (daily reset)
- Added `post_token_expiry_warnings()` method — runs at **23:00 EAT nightly**
- Delegates to `send_token_expiry_warnings()` in `interest_api_server.py`
- Feature-flagged via `TOKEN_EXPIRY_WARN_ENABLED` env var (default: enabled)

### 3. `send_token_expiry_warnings()` Worker (cis_controller/interest_api_server.py)

Scan logic:
- Reads all Sheets rows, checks Column Q (`COL_TOKEN_EXPIRY = 16`) for tokens expiring within `warn_within_hours` (default: 48h)
- Skips already-expired tokens (the expiry page handles those)
- Skips tokens outside the warning window
- Skips rows where `token_warning_sent = True` in PG (idempotency guard — prevents duplicate DMs)

Send paths:
- **Path A (Discord DM):** If `discord_id` is linked, sends KIRA DM: "Your enrollment link expires soon: [token URL]. Type `/renew` if it expires."
- **Path B (Brevo fallback):** If student hasn't joined Discord yet (no `discord_id`), sends Brevo HTML warning email with the same token URL
- After successful send (either path): sets `token_warning_sent = True` in PG + logs `token_expiry_warning_sent` event

Returns stats dict: `{scanned, warned_dm, warned_email, skipped_already_sent, skipped_no_token}`

### 4. Expiry HTML Page (cis_controller/interest_api_server.py → `_handle_mpesa_status`)

- `GET /api/mpesa-submit?token=expired` now returns an HTML page when the browser hits it directly (`Accept: text/html`)
- JS fetch calls (`Accept: */*`) continue to receive JSON 410 with `{"expired": true}` — form JS compatibility preserved
- HTML page text: "Your payment link has expired. In Discord, type `/renew` in a DM to KIRA, or visit **#help**."

### 5. `/renew` Slash Command (main.py)

- `@Facilitator`-role gated (same gate as `/recover-member`)
- Parameter: `email` (student's enrollment email)
- Flow:
  1. Scans Sheets to find student row by email
  2. Generates new token (`_generate_submit_token()`) + new 7-day expiry
  3. Writes to Column P (`COL_SUBMIT_TOKEN`) + Column Q (`COL_TOKEN_EXPIRY`) via `update_roster_cells()`
  4. Resets `token_warning_sent = False` in PG (resets warning cycle for new token)
  5. Sends fresh Brevo Email #2 (`send_enrollment_payment_email()`) with new URL
  6. Reports outcome to facilitator in ephemeral Discord message

## Gaps Found and Resolved

1. **Pre-Discord warning gap** (not in original spec): Spec assumed all students would have `discord_id` at warning time. Pre-Discord students (token issued but haven't joined yet) would have been silently skipped. **Fix:** Added Brevo fallback email path — all students with an expiring token get warned regardless of Discord join status.

2. **Content negotiation for expiry page**: The form JS (`enroll.js` in k2m-landing) parses JSON from the GET endpoint. Changing it to pure HTML would break form UX. **Fix:** Content negotiation via `Accept` header — HTML for browsers, JSON for JS fetches.
