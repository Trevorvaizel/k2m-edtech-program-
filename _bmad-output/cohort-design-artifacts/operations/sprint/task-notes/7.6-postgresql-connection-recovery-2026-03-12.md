# Task 7.6 Follow-up - PostgreSQL Connection Recovery Hardening (2026-03-12)

## Incident Trigger
- Repeated production dashboard alerts:
  - `PostgreSQL Connectivity Degraded`
  - `CIS Bot Health Check Failed` (Database)
- Health-monitor detail: `PostgreSQL connectivity check returned False`

## Production Evidence
- Railway logs showed:
  - `PostgreSQL connectivity check failed: SSL connection has been closed unexpectedly`
  - followed by repeated `PostgreSQL connectivity check failed: connection already closed`
- This confirms a dropped/stale psycopg2 connection was not being refreshed and the monitor remained in persistent degraded state.

## Root Cause
- `PgStudentStateStore.check_pg_connectivity()` used a single long-lived primary connection (`self.conn._conn`) and returned `False` on failure without restoring that primary connection.
- After one socket/SSL drop, subsequent checks kept hitting the same closed connection.

## Fix Implemented
- File: `cis-discord-bot/database/pg_store.py`
- Added reconnect-safe behavior in `PgConnectionWrapper`:
  - Detect closed connection before query and reacquire from pool.
  - Retry once after known transient connection-closed errors.
  - Return closed connections to pool as `close=True`.
- Added `PgStudentStateStore._reconnect_primary_connection()` helper.
- Upgraded `check_pg_connectivity()`:
  - Uses a pool probe connection for `SELECT 1`.
  - Refreshes primary connection if stale.
  - On probe failure, reconnects primary and retries once before reporting failure.

## Verification
- Added regression tests:
  - `cis-discord-bot/tests/test_pg_store_connectivity.py`
  - Covers:
    - closed primary connection refresh
    - SSL probe failure recovery
    - wrapper query retry after closed-connection error
- Test run:
  - `pytest -q tests/test_pg_store_connectivity.py tests/test_health_monitor.py tests/test_task_78_reliability.py`
  - Result: `37 passed`
