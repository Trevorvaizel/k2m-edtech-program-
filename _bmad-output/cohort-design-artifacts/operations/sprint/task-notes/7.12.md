## 2026-03-11 Production Hardening Addendum

### Incident observed during live join test

- A test member joined Discord and received the unmatched DM:
  - "It looks like you joined without a personal invite link..."
- The same test member could see broader channels than intended for `@Guest`.

### Root causes

1. **Permission boundary drift in server config**
   - `@Guest` explicit deny was not consistently present at category level across all content categories.
   - Result: a guest account could inherit broader visibility than the waiting-room contract.
2. **Invite-diff blind spot**
   - `on_member_join` invite matching could miss a one-use invite when it is created after snapshot and consumed/deleted before diffing.
   - Result: `student_unmatched` with `invite_code=None` even for a real join event.

### Production actions completed

- Applied category-level permission hardening:
  - `@Guest` deny mask (`view/send/read_history`) on all non-lounge categories.
  - Preserved explicit allow on `#welcome-lounge` for `@Guest`.
- Verified effective permission graph by API audit:
  - Non-lounge channels deny `@Guest`.
  - `#welcome-lounge` allows `@Guest`.
  - Audit result: `issues=0`.
- Deployed invite fallback patch to production (`main.py`):
  - When invite diff returns no code, bot now attempts a constrained fallback:
    - select a **unique** recent unlinked student candidate
    - use candidate `invite_code` for normal linking flow
  - Deployment completed successfully on Railway with healthy `/health` check.

### Verification

- Post-deploy startup logs confirmed normal boot + `Task 7.12: welcome-lounge initialized`.
- Internal API routes and health endpoint remained healthy after deployment.
- Behavior now matches waiting-room intent: `@Guest` should only operate in `#welcome-lounge` while awaiting full activation.

## 2026-03-11 Audit Remediation (Gap + Claim Validation Pass)

### Gaps found

1. **PostgreSQL runtime count gap**
   - `PgStudentStateStore` did not implement `get_confirmed_student_count()`.
   - Result: `#welcome-lounge` status refresh could report `0` confirmed in PG mode even when confirmed students existed.
2. **Permission drift hardening gap**
   - Runtime hardening for `@Guest` boundary needed stronger self-healing in case category/channel overwrites drift.
3. **Mixed-role edge case**
   - If `@Guest` removal failed during role upgrade, member could retain mixed `@Guest + @Student` state.

### Fixes applied

- Added `get_confirmed_student_count()` to `cis-discord-bot/database/pg_store.py`.
- Hardened `cis-discord-bot/cis_controller/welcome_lounge.py`:
  - Enforces `@Guest` deny on all non-lounge channels/categories during lounge upsert/refresh.
  - Treats any member with `@Student` as student in lounge handler (even if stale `@Guest` remains).
  - Attempts stale `@Guest` cleanup on student lounge messages.
- Hardened `cis-discord-bot/cis_controller/internal_api_server.py`:
  - If `@Guest` removal fails on role-upgrade webhook, applies member-level `#welcome-lounge` send deny fallback.
  - Response payload now includes `welcome_lounge_member_lock_applied`.
- Hardened `setup_discord_server.py`:
  - Guest base role set to neutral permissions (`Permissions.none()`).
  - Existing role permissions are reconciled on rerun.
  - Category-level guest deny baseline added; non-lounge channel guest deny is explicit.
