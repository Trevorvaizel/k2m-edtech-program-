## What Was Implemented

### 1) Shared `/health` endpoint on active HTTP server (Task 7.8 Part A)

Implemented in `cis-discord-bot/cis_controller/parent_unsubscribe_server.py`:
- Added `GET /health` route on the same shared port-8080 aiohttp app used in production.
- Health response contract:
  - `status`: `healthy` or `degraded`
  - `db`: `connected` or `disconnected`
  - `discord`: `connected` or `disconnected`
  - `details`: probe source details
  - `checked_at`: UTC timestamp
- HTTP status:
  - Always `200` for Railway liveness compatibility
  - Dependency state remains explicit in payload (`status`, `db`, `discord`)

DB probe logic:
- Uses PostgreSQL native check when available (`check_pg_connectivity()`).
- Falls back to lightweight `SELECT 1` on SQLite connection for local/test mode.

Discord probe logic:
- Uses the live bot reference mounted through internal/interest servers and checks `bot.is_ready()`.

### 2) Apps Script runtime error ingestion endpoint (Task 7.8 Part C)

Implemented in `cis-discord-bot/cis_controller/internal_api_server.py`:
- Added signed endpoint: `POST /api/internal/apps-script-error`
- Reuses existing HMAC signature guard (`X-K2M-Signature` via `require_webhook_signature`).
- Payload contract:
  - `type` (optional, when present must be `apps_script_error`)
  - `fn` or `function_name` (required)
  - `error` or `message` (required)
- Behavior:
  - Valid payload posts formatted alert to `#facilitator-dashboard` including function name, error text, source IP, and optional type.
  - Invalid signature returns `401` and logs 401 alert.
  - Invalid payload shape returns `400`.

Also updated route registration/mount logs so runtime now advertises:
- `/api/internal/role-upgrade`
- `/api/internal/preload-student`
- `/api/internal/apps-script-error`
