# Task 7.5: Payment Feedback DMs - Immediate + 24h Silence + Unverifiable Path

**Date:** 2026-03-10  
**Status:** done (audit-corrected)  
**Scope:** Decisions H-02, M-01, N-23, N-12

---

## Audit Outcome

Task 7.5 was implemented, but two gaps were found during this audit:

1. **Timing gap (H-02/N-23):** 24h silence and 8h escalation age was derived from lead `created_at` (Column N), not payment submission time.
2. **Validation gap (N-12):** `/api/mpesa-submit` accepted invalid code formats, increasing `Unverifiable` support noise.

Both were fixed and regression-tested.

---

## What Was Correct Before Audit

- Immediate fire-and-forget KIRA DM after successful `/api/mpesa-submit` (linked Discord users only).
- 6h scheduler pass wired at 00:05 / 06:05 / 12:05 / 18:05 EAT.
- Unverifiable student DM once + dashboard alert every pass.
- N-23 escalation to all members of `@Facilitator` role (guarded once per payment cycle).
- Bot injection via `InterestAPIServer.set_bot()` in `main.py`.

---

## Fixes Applied in This Audit

### 1) Accurate payment age anchor

- Added `payment_pending_since TIMESTAMP` to PostgreSQL schema.
- On successful `/api/mpesa-submit`, backend now sets `payment_pending_since = now(UTC)`.
- Scheduler worker (`send_payment_feedback_dms`) now computes age from `payment_pending_since`.
- For legacy rows without an anchor, worker initializes anchor once (to now) to avoid false overdue alerts from stale lead timestamps.

### 2) M-Pesa code format validation (N-12)

- Backend `/api/mpesa-submit` now enforces `^[A-Z0-9]{10}$`.
- Returns `400` with actionable message and example code when invalid.
- Frontend `k2m-landing/mpesa-submit` now enforces same 10-char constraint.

### 3) State reset on new payment cycle

On `/api/mpesa-submit`, backend now resets these flags for the student row:
- `payment_silence_dm_sent = false`
- `payment_escalation_dm_sent = false`
- `unverifiable_dm_sent = false`

This ensures each new payment attempt has a fresh DM/escalation lifecycle.

### 4) Dashboard channel env fallback

7.5 worker now resolves dashboard channel ID from:
- `CHANNEL_FACILITATOR_DASHBOARD` (primary)
- `FACILITATOR_DASHBOARD_CHANNEL_ID` (fallback)

## 2026-03-11 Production Hotfix (Runtime Store Reuse)

### Issue observed

During live smoke testing, `/api/mpesa-submit` intermittently hung from the client side and Discord heartbeat warnings appeared in logs (`heartbeat blocked`).

### Root cause

`interest_api_server.py` was creating a fresh store via `get_store()` inside async handlers/workers.  
In production this can instantiate `PgStudentStateStore` and run schema verification on the event loop thread, causing avoidable blocking under load.

### Fix applied

- Added `_resolve_store()` helper:
  - prefer `database.get_runtime_store()` (already initialized in `main.py`)
  - fallback to `get_store()` only when runtime store is unavailable (tests/standalone)
- Replaced direct `get_store()` usage in:
  - `send_token_expiry_warnings(...)`
  - `send_payment_feedback_dms(...)`
  - `/api/mpesa-submit` pending-flag update path

### Validation

  - No subsequent heartbeat-block warnings observed in the immediate post-fix log window.
