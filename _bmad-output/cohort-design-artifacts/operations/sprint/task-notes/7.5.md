## What Was Built

### 1. Immediate KIRA DM after M-Pesa submit (`interest_api_server.py`)

- After `_handle_mpesa_submit` successfully writes Column K (M-Pesa code) + Column L (Pending) and sends Brevo Email #3:
  - If student has a linked `discord_id` in Column D AND bot is available → fire-and-forget `asyncio.create_task()` sends KIRA DM
  - Message: *"Hey {name} — got your M-Pesa code! Trevor reviews within 24 hours. I will DM you when you are activated."*
  - Never blocks the 200 response — DM failure is logged as a warning, not surfaced to the student

### 2. Three PG Idempotency Flags (`database/schema_pg.sql` + `database/pg_store.py`)

New columns in `students` table (idempotent `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`):

| Column | Purpose |
|--------|---------|
| `payment_silence_dm_sent BOOLEAN DEFAULT FALSE` | Guards 24h silence DM — sent once per payment submission |
| `unverifiable_dm_sent BOOLEAN DEFAULT FALSE` | Guards "Unverifiable" student DM — sent once |
| `payment_escalation_dm_sent BOOLEAN DEFAULT FALSE` | Guards N-23 @Facilitator escalation — once per payment, reset if payment resolves |

Getter/setter methods added to `PgStudentStateStore` using shared `_get_bool_flag` / `_set_bool_flag` helpers (mirror of 7.4 `token_warning_sent` pattern). Placeholder-anchor upsert ensures idempotency for pre-Discord students.

### 3. `send_payment_feedback_dms()` Worker (`interest_api_server.py`)

Scans Google Sheets roster in a single pass. Three sub-checks per row:

**Pass A — 24h silence DM (Decision H-02):**
- Column L = `Pending` AND age (from Column N) > 24h AND student has `discord_id`
- Guarded by `payment_silence_dm_sent` PG flag (prevent repeat DMs)
- Message: *"Hey {name} — still reviewing your payment. If it's urgent, post in #help and a facilitator will assist."*

**Pass B — Unverifiable DM + dashboard alert (Decision M-01):**
- Column L = `Unverifiable`
- Student KIRA DM guarded by `unverifiable_dm_sent` flag (once)
- Dashboard alert posted to `#facilitator-dashboard` on every pass (🔴 alert always visible)
- Message: *"Hey {name} — we had trouble verifying your payment. Please send a screenshot of your M-Pesa confirmation to #help or DM Trevor directly."*

**Pass C — N-23 @Facilitator escalation (Decision N-23):**
- Column L = `Pending` AND age > 8h AND within business hours (08:00–20:00 EAT)
- Fetches all members with `FACILITATOR_ROLE_ID` from the Discord guild (Trevor + Bruce + Mkunzi)
- DMs ALL @Facilitator members once (guarded by `payment_escalation_dm_sent` flag)
- Message: *"⚠️ Payment pending >8h — {name} ({email}). Age: {n}h. Please verify in Google Sheets."*

### 4. 6h Scheduler Pass (`scheduler/scheduler.py`)

- New state var: `_payment_silence_check_last_run`
- Fires at **00:05, 06:05, 12:05, 18:05 EAT** (6h bucket: `hour % 6 == 0 AND minute == 5`)
- Calls `post_payment_silence_check()` which delegates to `send_payment_feedback_dms()`
- Feature-controlled by `GOOGLE_SHEETS_ID` presence (no-ops gracefully if missing)

### 5. Bot Injection into `InterestAPIServer` (`main.py`)

- `InterestAPIServer.__init__` now accepts `bot=None` param + `set_bot()` method (mirrors `InternalWebhookServer` pattern)
- `main.py` `on_ready` calls `interest_api_server.set_bot(bot)` after the server is created

## Gaps Found and Resolved

1. **Bot reference gap in InterestAPIServer**: The immediate DM requires a Discord bot reference. `InterestAPIServer` had no bot injection mechanism. **Fix:** Added `bot=None` param + `set_bot()` on the class; wired in `main.py` `on_ready`.

2. **Pre-Discord student coverage**: The immediate post-submit DM only fires for students who have already joined Discord (have a `discord_id`). Students who submit M-Pesa before joining Discord get the Brevo Email #3 confirmation. This is correct behaviour per Decision B-03 (KIRA DMs only after `on_member_join`).

3. **Dashboard alert frequency design**: Unverifiable dashboard alerts fire on every 6h pass (not just once) so the alert remains prominent until resolved. The student DM is guarded (once only). This was an intentional deviation from a naive idempotent-everything approach.

4. **N-23 escalation reset policy**: Per Winston + Dev discussion, `payment_escalation_dm_sent` is set to `True` once and NOT auto-reset. Trevor or a facilitator resolves the payment (changes Column L), at which point the flag can be reset via future tooling or manual PG update. This prevents facilitators being DM'd repeatedly for a 3-day stale payment.
